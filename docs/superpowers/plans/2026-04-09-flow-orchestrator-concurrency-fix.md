# FlowOrchestrator Concurrency Fix Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Prevent concurrent `run()` calls from executing flow steps in parallel by adding a pessimistic database lock and a `NOT_STARTED` step status.

**Architecture:** Wrap `run()` in a Prisma interactive transaction with `SELECT ... FOR UPDATE SKIP LOCKED` on the FlowInstance row. Add `NOT_STARTED` enum value so only the first step starts as `READY`; subsequent steps are promoted after the previous step succeeds.

**Tech Stack:** Prisma (PostgreSQL), TypeScript, Jest

**Spec:** `docs/superpowers/specs/2026-04-06-flow-orchestrator-concurrency-fix-design.md`

- [ ] **Step 1: Wrap `run()` in a transaction with `FOR UPDATE SKIP LOCKED`**

Replace the `run()` method in `FlowOrchestrator.ts`:

```typescript
public async run(flowInstanceId: string): Promise<void> {
  const prisma = await this.dbProvider.getClient()

  await prisma.$transaction(async (tx) => {
    const locked: Array<{ id: string }> = await tx.$queryRawUnsafe(
      'SELECT id FROM "FlowInstance" WHERE id = $1 FOR UPDATE SKIP LOCKED',
      flowInstanceId,
    )

    if (locked.length === 0) {
      this.logger.info('Flow instance locked by another caller; skipping', { flowInstanceId })
      return
    }

    await tx.flowInstance.updateMany({
      data: { status: FlowInstanceStatus.IN_PROGRESS },
      where: { id: flowInstanceId, status: FlowInstanceStatus.WAITING },
    })
    let claimed = await this.claimNextStep(tx, flowInstanceId)

    while (claimed) {
      const current = claimed
      const runtime = await this.buildRuntimeContext(flowInstanceId, tx)
      const snapshot = this.readSnapshot(runtime.flowSnapshot)
      const stepDefinition = snapshot.steps.find(step => step.stepOrder === current.stepOrder)
      if (!stepDefinition) {
        await tx.flowStepInstance.update({
          data: { endedAt: new Date(), error: { message: 'Step definition missing' }, status: FlowStepStatus.FAILED },
          where: { id: current.id },
        })
        await tx.flowInstance.update({
          data: { status: FlowInstanceStatus.FAILED },
          where: { id: flowInstanceId },
        })
        return
      }

      const executor = this.executorRegistry.get(current.stepType)
      await tx.flowStepInstance.update({
        data: {
          input: this.normalizeJson({
            config: stepDefinition.config,
            context: runtime.context,
          }),
        },
        where: { id: current.id },
      })
      const result = await executor.execute({
        config: stepDefinition.config,
        runtime,
        stepOrder: current.stepOrder,
      })

      await this.persistStepOutcome(current.id, result, tx)

      if (result.outcome === 'waiting' || result.outcome === 'failed') {
        await tx.flowInstance.update({
          data: {
            status: result.outcome === 'waiting' ? FlowInstanceStatus.WAITING : FlowInstanceStatus.FAILED,
          },
          where: { id: flowInstanceId },
        })
        return
      }

      const nextOrder = this.resolveNextOrder(snapshot.steps, current.stepOrder)
      if (nextOrder !== null) {
        await tx.flowStepInstance.updateMany({
          data: { status: FlowStepStatus.READY },
          where: {
            flowInstanceId,
            status: FlowStepStatus.NOT_STARTED,
            stepOrder: nextOrder,
          },
        })
      }
      await tx.flowInstance.update({
        data: { currentStepOrder: nextOrder },
        where: { id: flowInstanceId },
      })

      claimed = await this.claimNextStep(tx, flowInstanceId)
    }

    await tx.flowInstance.update({
      data: { status: FlowInstanceStatus.COMPLETED },
      where: { id: flowInstanceId },
    })
  }, { timeout: 120000 })
}
```

- [ ] **Step 3: Verify handleSignal path is correct**

`handleSignal()` calls `persistStepOutcome()` directly (without a tx client) for signal-completed steps, then calls `run()` which acquires its own lock. This is correct because:
- The signal step update is idempotent (it transitions a WAITING step)
- `run()` will serialize with any other concurrent caller via the lock
